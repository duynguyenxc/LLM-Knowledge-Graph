# Architecture FAQ: Understanding the System's Outputs

This document answers two common questions about how the pipeline works and how all the output files relate to each other.

---

## Q1. What is the Knowledge Dashboard HTML for, and how is it different from the data files in `graphrag_data/`?

**Short answer:** They contain the same information — the HTML file is just the human-readable version of the raw data files.

### The Full Picture

When `python -m graphrag index --root ./graphrag` finishes running, it saves everything it learned about the 28 academic papers into a set of structured database files inside `outputs/graphrag_data/`:

| File | What it contains | Who can read it |
|---|---|---|
| `entities.parquet` | A database table of all 1,318 extracted concepts, each with a name, CMO type, and full academic description | Python scripts, AI agents |
| `relationships.parquet` | A database table of all 1,672 causal connections between those concepts | Python scripts, AI agents |
| `community_reports.parquet` | 245 AI-written paragraph summaries, one per semantic cluster of related concepts | Python scripts, AI agents |
| `graph.graphml` | The full network map encoded in a standard XML format (readable by tools like Gephi, Neo4j, NetworkX) | Specialized graph tools |

These files are the **raw "brain"** of the system. They are precise and machine-friendly, but completely unreadable to a human researcher at a glance.

The `Knowledge_Dashboard.html` file solves this problem. It was generated by running `python scripts/visualize_dashboard.py`, which reads `entities.parquet` and `graph.graphml`, and translates all that raw data into a visual, interactive web interface.

```
entities.parquet   ─────┐
                        ├──► visualize_dashboard.py ──► Knowledge_Dashboard.html
graph.graphml      ─────┘
```

Think of it this way:

> `graphrag_data/` is the **raw brain** — precise, complete, but unreadable to humans.  
> `Knowledge_Dashboard.html` is the **screen that displays the brain** — the same data, made visible and explorable.

They are **not different data**. The HTML file simply re-renders the `.parquet` and `.graphml` content as colored dots, connecting lines, a search bar, and a side panel so that a researcher can click around and explore the literature map intuitively.

## Q2. I ran multiple commands — what exactly did each one produce?

There are **three commands** in this project, each serving a completely separate purpose and producing different outputs.

### Command 1 — Build the Knowledge Graph
```bash
python -m graphrag index --root ./graphrag
```

| | Detail |
|---|---|
| **Reads from** | `graphrag/input/` — the 28 plain-text paper files |
| **Does** | Runs GPT-4o-mini across all papers to extract CMO entities, maps relationships, runs Leiden clustering to form 245 semantic communities, and generates vector embeddings |
| **Produces** | `outputs/graphrag_data/` — all the `.parquet` databases and `graph.graphml` |
| **HTML output?** | ❌ None |

---

### Command 2 — Visualize the Knowledge Graph
```bash
python scripts/visualize_dashboard.py
```

| | Detail |
|---|---|
| **Reads from** | `outputs/graphrag_data/entities.parquet` + `graph.graphml` |
| **Does** | Filters to the 300 most-connected core concepts, builds a physics-based network layout, and writes a self-contained HTML file with the `vis.js` library embedded |
| **Produces** | `outputs/Knowledge_Dashboard.html` — the interactive split-screen explorer |
| **Purpose** | Gives human researchers a way to visually explore and validate the Knowledge Graph before trusting the AI agents to synthesize it |

---

### Command 3 — Run the 10-Agent Systematic Review
```bash
python main.py
```

| | Detail |
|---|---|
| **Reads from** | `graphrag/input/` — the same raw paper files (not the `.parquet` files) |
| **Does** | Runs the full LangGraph state machine: generates the Initial Programme Theory, builds the study registry (S001–S028), performs two-stage PRISMA screening, extracts one primary CMOC per paper, checks for contradictions, synthesises the Programme Theory, and generates all academic report files |
| **Produces** | `outputs/PRISMA_report.md`, `outputs/evidence_table.md/.json`, `outputs/programme_theory_final.md`, `outputs/audit_trail.md` |
| **Purpose** | This is the **academic output** — what a researcher would submit as the findings of the systematic review |

---

### Summary: The Complete Pipeline at a Glance

```
graphrag/input/    (28 paper .txt files)
        │
        │  Step 1: python -m graphrag index --root ./graphrag
        ▼
outputs/graphrag_data/           ← Raw Knowledge Graph (machine-readable)
  ├── entities.parquet
  ├── relationships.parquet
  ├── community_reports.parquet
  └── graph.graphml
        │
        │  Step 2: python scripts/visualize_dashboard.py
        ▼
outputs/Knowledge_Dashboard.html ← Visual map (human-readable browser UI)

        │
        │  Step 3: python main.py  (reads graphrag/input/ directly)
        ▼
outputs/                         ← Academic report artifacts
  ├── PRISMA_report.md
  ├── evidence_table.md / .json
  ├── programme_theory_final.md
  └── audit_trail.md
```

> **Important note:** Commands 2 and 3 are completely independent of each other.  
> The Dashboard (Step 2) is for **human exploration and validation** of the raw graph.  
> The Agent pipeline (Step 3) is for **generating the academic deliverables** — PRISMA flow, CMOC table, Programme Theory — that get submitted to the journal.
